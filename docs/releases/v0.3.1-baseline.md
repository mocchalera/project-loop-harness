# v0.3.1 implementation baseline

This document freezes the shipped v0.3.1 behavior as a reproducible comparison
baseline. It does not certify every observed behavior as desirable. Follow-on
MCP and transactional-outbox work may intentionally change these contracts.

## Release identity

- Baseline commit: `e5a7cadeff867b2c73587295dfe9c05d9675c568`
  (`main` after the v0.3.1 release record and integrated-roadmap adoption).
- Release tag: `v0.3.1`, pointing to release commit
  `eb6982444a01cf68c7308c48621b1704d1e5881f`.
- Package version: `0.3.1`.
- Schema version: `7`.
- Supported Python versions: 3.10, 3.11, 3.12, and 3.13. The package requires
  Python `>=3.10`, and the v0.3.1 release CI matrix covered those four versions.

## Migration baseline

The ordered migration list is:

1. `001_initial`
2. `002_tasks`
3. `003_agent_registry`
4. `004_code_index`
5. `005_verification_feedback`
6. `006_evidence_task_link`

v0.3.1 made no schema change from v0.3.0. The historical fixture at
`tests/fixtures/v0.3.1-baseline/db/v0.3.0-schema-7.sqlite3` therefore exercises
opening, inspecting, and applying the current migration command as an
intentional no-op. It also verifies that the database stays consistent at
schema 7 and contains no real project rows.

## CLI snapshot baseline

The single canonical fixture directory is
`tests/fixtures/v0.3.1-baseline/`. Its README defines the reuse contract for
later schema and CLI contract tests. Snapshots cover:

- `pcl --version`, `pcl --help`, and help for `validate`, `render`, `next`, and
  `context check`;
- `validate --strict --json`, `render --json`, `next --json`, and
  `context check --json` on a freshly initialized empty project;
- the same JSON commands on a synthetic representative project.

The generator normalizes timestamps, absolute project paths, UUIDs, and
40/64-character hexadecimal hashes. The reproducibility test generates the
whole snapshot set twice and requires byte-for-byte equality before comparing
it with the committed baseline.

## Known limitations

- Snapshot normalization can hide a change that affects only a normalized
  timestamp, path, UUID, or hash value. Field presence and surrounding payload
  structure remain covered.
- The representative fixture is synthetic and intentionally small; it does not
  model every workflow, error, or human-gate state.
- `context check` requires a task or job target. The empty-project snapshot
  records the typed missing-target error for `T-0001`; the representative
  snapshot records an existing task without a bound receipt.
- Because v0.3.0 and v0.3.1 both use schema 7, the database smoke does not
  exercise new DDL. A future schema version must add a contract test that copies
  this frozen DB and applies the new migration.
- This task does not correct unstable runtime output, MCP stdio framing, or
  event/JSONL dual-write behavior; those are follow-on work.
