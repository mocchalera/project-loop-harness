# Dashboard Design

## Purpose

The dashboard should answer three human questions:

1. Where is the loop right now?
2. What is risky or blocked?
3. What human decision is needed next?

## Required panels

Milestone 1:

- counts: features, open defects, goals, decisions, workflow runs;
- features table;
- defects table;
- goals table.

Milestone 2:

- current goal;
- active workflow;
- active agent jobs;
- verification results;
- stop conditions;
- risk and blocker summary;
- escalation queue;
- budget usage;
- recent events.

Current additive panels:

- needs-your-decision queue from `human_decisions`;
- story coverage and test coverage tables;
- task backlog with dependency and dependent ids;
- workflow proposal review queue;
- reports table with derived evidence/job/run links.

## Hard rules

- Dashboard must be generated by `pcl render`.
- Dashboard must be deterministic.
- `.project-loop/dashboard/dashboard-data.json` must follow the versioned contract in [dashboard-data-contract.md](dashboard-data-contract.md).
- Dashboard must link to evidence paths where possible.
- Dashboard must warn if validation fails.
- Dashboard must never become the source of truth.
