## Part A: 俯瞰 — v0.2 は何を達成するリリースか

### 北極星との接続

「文脈の領収書が作れる CLI」→「AI 開発の文脈制御盤」の 3 本柱のうち、
v0.1.10〜v0.1.12 は Trust と Integration を固めた。v0.2 は **Judgment を
支える計測** に入る。PLH 自身が出した検証提案について、

- 実行されたか / skip されたか / どんな evidence が残ったか
- fixture が dogfood から増えているか

を後から検査できる形で測れるようになること。これが v0.2 の成功条件
（レビュー議題 問い5）。機能の数ではない。

### 現在地の事実

- v0.1.12 は PyPI リリース済み（385 tests green、schema v4）。
- レビュー議題の P0/P1 三点 — sdist 自己完結（0085 相当）、
  `pcl verification feedback` への命名統一（0086 相当）、
  `refresh_replay` scope fidelity（0091 相当）— は main の 204a857
  で対応済み。**ただし agent-tasks 起票なし・TASKS.md 未記載・
  リリース未搭載**。次のリリースに乗って初めて PyPI 上の問題が解消する。
- v0.2 設計（migration 005 / suggestion ID / dogfood-to-fixture /
  advisory gate）は坂本承認済み。migration 005 の
  `require_human_approval: database_migration` 承認も取得済み。
- 実装順も承認済み: 0087 → 0088 → 0089 → 0090。

### リリース戦略（提案）

**推奨: 2 段リリース。**

| リリース | 内容 | 理由 |
|---|---|---|
| **v0.2.0** | M0 + 0087 + 0088（suggestion ID、migration 005、`pcl verification feedback`、observable rates） | schema migration を含むリリースは小さく観測可能に保つ。記録機能が先に出れば、0089/0090 を実装している間に **実 dogfood の feedback 行が貯まり**、fixture 提案と baseline が最初から実データで検証できる。sdist 修正も最速で PyPI に届く。 |
| **v0.2.1** | 0089 + 0090（dogfood-to-fixture、baseline record/compare、CI advisory 比較） | 計測基盤は feedback データがある状態で出す方が、design の「dogfood に食わせてから baseline」という順序思想に一致する。 |

代替案は 0087〜0090 を一括で v0.2.0 にする単発リリース。ループ全体が
一度に閉じる利点はあるが、migration 入りリリースが太くなり、baseline が
synthetic データだけで初期化されるため推奨しない。

### M1: 0087 — Verification Suggestion IDs

スコープ（設計承認済み、migration なし）:

- `code_context/receipts.py`: `verification_suggestions` を
  string-list から object-list `{id, command, reason}` へ。
  ID は `<receipt evidence_id>/VS-<nn>`（receipt 内 ordinal、決定的）。
- `code_context/summary.py`: 両形式受理（旧 receipt は `id: null` で
  要約）。summary JSON に `id` を含める。
- `receipt_show.py` / context pack 表示: 人間向け表示は command 中心で
  大きく変えない。
- receipt に `status` 字段は絶対に置かない（不変の候補提示）。

受け入れ条件:

- 新旧両形式の receipt fixture で summary / `pcl receipt show` /
  `context pack --include-code-context` が green。
- ID の決定性テスト（同一 receipt → 同一 ID 列）。
- 契約文書（`docs/code-context.md`、`docs/context-pack.md`）更新。

リスク: 低。絶縁層（`code-context-summary/v0`）が
`context-pack/v1` 消費者を守る — この目的のために作った層。

### M2: 0088 — Migration 005 + `pcl verification feedback`

スコープ（設計承認済み）:

- `src/pcl/db/`（SQL 発見方式）に migration 005 追加:
  `verification_feedback` append-only event table。
  `UNIQUE(suggestion_id)` なし、`executed` は `result` +
  `supporting_evidence_id` 必須の CHECK、evidence への FK
  （`PRAGMA foreign_keys = ON` 済みなので実効）。SCHEMA_VERSION 4→5。
- `cli.py` の既存 `verification` パーサ（cli.py:621）配下に
  `feedback` サブコマンド。`verify` alias は作らない。
- Referential honesty: suggestion ID の prefix から receipt evidence を
  ロードし、ID 実在を検証。不在は typed error。
- 挿入ごとに JSONL event append（既存 events 経路）。
- Observable rates: `execution_rate` / `executed_pass_rate` /
  `executed_fail_rate` / `feedback_coverage_rate` のみ。
  hit/miss 語彙は禁止。露出面は Part D-2 の決定に従う。
- 「no feedback recorded」は派生表示であり保存しない。

受け入れ条件:

- migration up テスト + 0076 系整合（never-downgrade、旧バイナリの
  DB-ahead typed 拒否が v5 で機能することを明示テスト）。
- CHECK 制約の境界テスト（executed で result 欠落 → 拒否、等）。
- 不在 suggestion ID → typed error テスト。
- 同一 suggestion への複数 feedback 行が挿入でき、"latest" が
  読み取り時派生であることのテスト。
- strict validate が feedback → evidence 参照の整合を検査する。
- dogfood: PLH 自身の実 receipt の実 suggestion に feedback を最低
  数件記録し、evidence 添付で通ることを確認。

リスク:

- **スキーマは一度出すと引っ込められない。** 出荷前に語彙
  （status/result の enum）を最終確認する。設計承認済みだが、
  spec 起票時に design doc と一字一句照合する。
- epistemic 境界: `executed + passed` は記録者の申告であり PLH の
  検証事実ではない。CLI ヘルプ・docs・event 語彙すべてで
  「claim + evidence pointer」の言い回しを守る（検収の重点項目）。

### M4: 0089 — Dogfood-to-fixture Propose

スコープ（設計承認済み）:

- `pcl eval fixture propose --from-receipt <evidence-id>` が
  `retrieval-fixture/v0` 候補を生成。changed files と retrieved
  candidates は実 receipt 由来、`expected_files` / `expected_tests` /
  `critical_context` は**空**、`labels_status: "unlabeled"`。
- staging は `fixtures/proposed/`（配置は Part D-3 で確定）。
  `tests/fixtures/` への採用は人間ラベル後の手動 move のみ。
  auto-adoption 経路は作らない。

受け入れ条件:

- 生成候補が可視的に unlabeled であること（ground-truth 捏造ゼロ）の
  契約テスト。
- v0.2.0 dogfood で貯まった実 receipt から propose を実走し、
  1 件以上を人間ラベル → `tests/fixtures/` 採用まで通す
  （ワークフロー全体の実証）。

### M5: 0090 — Baseline Record / Compare

スコープ（設計承認済み）:

- `pcl eval retrieval --record-baseline`: 結果を evidence
  （`.project-loop/evidence/retrieval-eval/`）+ event 化。
  provenance 全 6 項目必須: fixture path+content hash / git HEAD /
  index run id+detail hash / code_context config hash / pcl version /
  eval contract version。
- `--compare-baseline`: 同一 fixture hash の最新 baseline と
  per-metric delta を報告。
- CI: advisory 比較 step。metric threshold は advisory、
  eval 基盤破損（schema 破損・コマンド失敗・fixture 契約違反）のみ
  blocking。
- fixture 5 種（code change / docs-only / config-only / rename-move /
  secret omission）を揃える。0080 の adversarial 3 種 + 既存を棚卸し、
  不足分のみ追加。

受け入れ条件:

- provenance 欠落 baseline は typed error。
- fixture hash 不一致の compare は「比較不能」を明示（黙って比較しない）。
- CI で advisory 比較が artifact として残り、metric 悪化で CI が
  落ちないこと・基盤破損で落ちることの両方をテスト。

### M6: v0.2.1 リリースゲート

M3 と同一の機械ゲート + baseline 履歴が 2 点以上（比較が実際に
機能した証拠）+ dogfood 由来 fixture が 1 件以上採用済み。

## Part C: 専門家視点 — 分野別リスクレビュー

### スキーマ / DB 整合性

- FK は `PRAGMA foreign_keys = ON`（db.py:14）で実効。
  `verification_feedback` → `evidence` の FK は evidence 削除経路が
  ないため実質 append-only 同士で安全。
- CHECK 制約は SQLite で挿入時に実効するが、**CLI 層でも同じ検証を
  typed error として先に行う**こと（DB エラーをユーザーに漏らさない
  のが既存流儀）。
- SCHEMA_VERSION 4→5 は 0076 の never-downgrade / DB-ahead 拒否と
  組で初めて安全。旧バイナリ試験を M3 ゲートに含めた理由。

### CLI 契約 / 命名

- `verification feedback` は既存 `verification` namespace への追加で
  確定（204a857 で design doc 修正済み）。`verify` alias は
  「意味なく併存」を避けるため v0.2 では作らない。
- suggestion ID `E-xxxx/VS-nn` はスラッシュを含む — shell 引用の
  注意を CLI ヘルプに一行入れる（`--suggestion 'E-0001/VS-01'`）。
- 旧 receipt（string-list）は永続的に受理。summary の後方互換テストを
  fixture として固定（`tests/fixtures/` に旧形式サンプルを恒久保存）。

### 評価・計測

- **`execution_rate` の分母定義が唯一の未固定点**（design doc 自身が
  「definition fixed in the task spec」としている）。0088 spec 起票時に
  確定必須 → Part D-2。曖昧なまま worker に渡すと 0078 の再演になる。
- metric は observable rates のみ。usefulness は計算しない・
  hit/miss と呼ばない — 検収時の語彙チェック項目に追加。
- baseline の provenance 6 項目は「1 つでも欠けたら比較不能」が正しい
  振る舞い。advisory はあくまで metric threshold の話で、
  provenance 完全性は契約（blocking）。

### 配布・リリース

- sdist 自己完結は 204a857 の MANIFEST.in +
  `scripts/verify_sdist_contracts.py` + CI step で解決済みだが、
  **PyPI 上の 0.1.12 sdist は壊れたまま**。v0.2.0 を早く出すことが
  そのまま修正になる（2 段リリース推奨の一因）。
- 0.1.x → 0.2.0 のバージョン跳躍は「migration 005 + 契約追加」の
  シグナルとして妥当。README の contract 一覧に
  `verification-feedback` 系の記載を追加。

### 運用（worker 委譲）

- 確立済みプロセスを維持: spec 先コミット → codex worker
  （cockpit、worktree、`--worktree` ブランチ = agent-tasks/00xx）→
  Fable 独立検収（determinism / epistemic 語彙 / 敵対的ケース）→
  main マージ。
- 既知の罠は解消・文書化済み: pytest `pythonpath=["src"]` により
  stale-import 罠は恒久解消。worker の editable install 付け替えのみ
  検収後に canonical repo で `pip install -e .` し直す。
- 並行実行はしない。0087→0088 は直列（0088 が 0087 の ID 契約に依存）、
  0089→0090 も直列（設計承認済みの順序）。

## Part D: 決定事項（2026-07-06 坂本承認）

1. **204a857 の事後起票**: 0085/0086/0091 を簡潔な事後スペックとして
   起票する。各スペックに implemented_in（`204a857`）、why、
   verification を残す。大きな再設計文書は作らない。→ 起票済み。
2. **metrics 定義と露出面**:
   - 露出は read-only の `pcl verification stats --json` から開始。
   - 分母は「addressable な発行済み suggestion」（object 形式で
     `id` が非 null のもの）。旧 receipt の `id: null` suggestion は
     分母に入れず `unaddressable_legacy_suggestions_count` として
     別会計。
   - `feedback_coverage_rate` / `execution_rate` は suggestion 単位、
     `executed_pass_rate` / `executed_fail_rate` は feedback event
     単位。空分母は `null`。
   - hit/miss/usefulness 語彙は引き続き禁止。
   → 0088 spec と design doc の承認追補に反映済み。
3. **fixture staging**: リポジトリ直下 `fixtures/proposed/` を採用。
   `.project-loop/` ではなくレビュー対象のソース資産として Git で
   扱う。proposed fixture は必ず `labels_status: "unlabeled"` で
   expected/critical_context は空のまま。
4. **リリース戦略**: 2 段で確定。v0.2.0 = M0 + 0087 + 0088、
   v0.2.1 = 0089 + 0090。migration 入りの v0.2.0 は小さく切り、
   dogfood feedback を貯めてから fixture/baseline へ進む。
5. **追加指示（version truthfulness）**: migration metadata の
   `pcl_version` が `"0.1.0"` 固定だった問題を v0.2.0 出荷前に修正
   （`migrations.py` が `pcl.__version__` を刻印、テスト追加）。
   → 修正済み。

## 変更しないこと（再確認）

semantic promotion gate は不変: embeddings / Tree-sitter / call graph /
semantic retrieval は、5 種 fixture での lexical チューニング後も
missing-critical-context が改善しない場合のみ。自動実行・go/no-go 判定・
content-based secret scanning・hosted 系はすべて引き続き対象外。
