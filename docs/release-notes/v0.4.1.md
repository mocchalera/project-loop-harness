# project-loop-harness v0.4.1

**Prepared:** 2026-07-11
**Status:** Local release candidate; not published
**Scope:** Integrity Migration

This release completes the advisory-to-enforced lifecycle-integrity migration
without a database migration, dependency addition, or automatic semantic
decision. Existing projects remain advisory until an operator explicitly
enables enforcement; newly initialized projects continue to default to
enforced lifecycle validation.

## Highlights

### Neutral idle routing (0141)

- `pcl next` now returns a factual `idle` action instead of fabricating a human
  decision when no work is pending.
- Explicit new intent continues through `pcl start`; idle inspection remains
  read-only.

### Plan-first lifecycle repair (0142 / 0143)

- `pcl repair lifecycle` produces a deterministic, read-only action model with
  structural, semantic, human-review, and unsupported classifications.
- Dedicated audited link commands repair Test/Story/Evidence relationships.
- `--apply-structural` applies only planner-selected structural actions and is
  an event-free no-op when no safe structural action exists.
- Story approval, Evidence interpretation, Goal proof, and other semantic or
  human decisions remain explicit operator actions.

### Skill/runtime provenance (0144)

- `pcl start` with explicit repeatable `--skill` records canonical
  `execution-provenance/v1` artifacts as existing schema-8 Evidence.
- The matching event anchors the artifact SHA-256 before current Skill hashes
  are inspected, making recorded execution provenance independently checkable.
- `pcl finish` discovers and reports existing Task provenance; it does not
  record a new provenance artifact.

### Structured validation diagnostics (0145)

- JSON validation output now includes additive structured findings while
  retaining the existing string `errors` and `warnings` fields.
- Findings expose stable code, severity, entity, related-entity, and repair
  metadata, including safe inspection or repair commands where supported.
- Unsupported and human-review findings remain honest stops rather than being
  silently reclassified as automatically repairable.

## Migration dogfood (0145a)

The complete path was exercised against a disposable project created by the
released v0.3.0 source at commit
`f04d9f70394eb288cfdb5dd2cd1bedef3f07c96c`. Current code migrated it from
schema 7 to schema 8, preserved advisory policy, produced byte-identical
read-only repair plans, executed emitted inspection commands, refused to invent
unsupported Evidence meaning, and performed an event-free structural no-op.

An operator then reviewed and approved the Story, copied hash-bound Evidence,
relinked the Test explicitly, and enabled enforcement. Final strict validation
returned no errors, warnings, or findings; audit reported 19 matching SQLite,
JSONL, and delivered-outbox events with zero anomalies; render succeeded. The
automated regression freezes that sequence. Full commands, hashes, and the
known unsupported cases are recorded in
`docs/dogfood-report-v0.4.1-integrity-migration.md`.

## Compatibility and boundaries

- DB schema remains **8**; there is no migration in v0.4.1.
- Runtime dependencies remain empty.
- `completion-packet/v1` and `handoff-packet/v1` are unchanged.
- Fresh projects continue to use enforced lifecycle validation by default.
- Existing projects without `validation.lifecycle_integrity` remain advisory
  until an operator explicitly enables enforcement.
- No hosted backend, cloud sync, telemetry, paid service, or automatic agent
  launch is added.
- Package/MCP version metadata changes from 0.4.0 to 0.4.1. The independently
  versioned Codex plugin manifest is unchanged.
- Task IDs shipped: 0141, 0142, 0143, 0144, 0145, and 0145a. Task 0145b is the
  local release-preparation gate.

## Known advisory findings and residual risk

The representative migration intentionally leaves legacy states such as
conflicting links, drifted artifacts, invalid Goal proof, and open Defects in
unsupported or human-review classifications. v0.4.1 does not claim automatic
repair for those states. Operators must review every semantic action and should
retain backups before migrating valuable project state.
