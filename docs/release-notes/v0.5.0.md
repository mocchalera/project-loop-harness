# project-loop-harness v0.5.0

**Prepared:** 2026-07-14

**Status:** Local release candidate; not published

**Scope:** Council Profile + Adoption / Distribution

This release adds an opt-in, contract-first discovery Profile while making the
local control plane easier to adopt, supervise, and recover. Direct execution
remains the default for clear tasks. Council output remains advisory Evidence;
humans retain every selection, authorization, and publication decision.

## Highlights

### Opt-in Council Discovery Profile

- Packaged standard-library contracts validate profile manifests, run
  requests, output bundles, claim sets, verification plans, decision proposals,
  and authorization provenance.
- Read-only request preparation and dry-run bundle planning precede a single
  atomic ingest boundary with idempotency and audit diagnostics.
- The built-in offline fixture proves source, wheel, and sdist behavior without
  a provider, network request, paid service, or embedded model runner.
- Decision projection and paid/network authorization stay hash-bound and
  human-controlled. The evaluation outcome is `continue experiment`, not
  adopt-by-default.

### Adoption and operator experience

- README and adoption guidance lead with a 30-second value path, five-minute
  setup, inspect-first coexistence, and the agent-owned routine loop.
- The alpha stability policy defines protected JSON/error/migration surfaces
  and explicitly unstable internals.
- The Japanese dashboard starts with `今 / 完了 / 次 / あなたの判断 / 注意点`
  and preserves advanced evidence/state panels under progressive disclosure.
- Dashboard decisions name the pending action and authority without approving
  human gates or changing the stable `dashboard-data/v1` contract.

### Field-feedback and recovery loop

- `pcl finish` diagnoses missing checks, copied Evidence, completion-policy
  receipts, timeouts, and recovery commands without manufacturing approval.
- `pcl guide` exposes structured purpose-oriented start/direct/finish/dashboard
  and recovery routes.
- `pcl report skill-usage` locally summarizes privacy-safe command/friction
  signals, distinguishes explicit exit status, and avoids treating successful
  composite output or its own report as a product failure.

## Compatibility and boundaries

- DB schema remains **8**; there is no migration in v0.5.0.
- Runtime dependencies remain empty and Python support remains 3.10–3.13.
- Versioned public contracts remain additive unless explicitly documented.
- Existing v0.4.x projects remain inspect-first migration/adoption inputs.
- The Codex plugin manifest remains independently versioned.
- No hosted backend, cloud sync, telemetry, production database access,
  automatic GitHub write, real-provider execution, or paid-service dependency
  is added.
- Task IDs shipped: 0154–0160, 0162–0173. Task 0161 remains parked after the
  offline dogfood slice because second-repository and real-provider work are
  separately human-gated.
