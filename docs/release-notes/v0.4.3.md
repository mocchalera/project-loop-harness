# project-loop-harness v0.4.3

**Prepared:** 2026-07-11
**Status:** Local release candidate; not published
**Scope:** Evidence Completeness

This release makes terminal state depend on a complete, target-bound evidence
set instead of a single selected positive report. It adds deterministic policy
evaluation for external verdicts, keeps unfinished passing work actionable, and
records approval provenance without allowing an agent to manufacture human
authority.

## Highlights

### Complete Evidence Sets

- `evidence-set/v1` records the target, explicit work root and report manifest,
  included Evidence, required report kinds, and known exclusions.
- Missing or known-failing required reports fail closed.
- Planning and inspection are read-only; recording is an explicit mutation.

### Deterministic Completion Policy

- `completion-policy/v1` evaluates allowlisted JSON predicates without running
  third-party code or calling a model.
- A `prototype` verdict cannot satisfy a policy requiring `complete`.
- Terminal Test transitions require the hash-bound evidence-set and policy
  receipts selected for that acceptance check.

### Actionable Routing and Approval Provenance

- `pcl next --json` continues to route passing-but-unfinished Features instead
  of reporting an idle loop.
- Approval receipts distinguish actor, recorder, and source. Agent-mediated
  human approval requires a factual conversation or Cockpit source reference.
- Bundled `project-control-loop` Skill copies use Evidence-ID-first terminal
  commands and automatically continue in-scope agent-safe work while reporting
  Now, Done, Next, and Human needed.

## Dogfood

Canonical incomplete and complete cross-skill fixtures exercise the LP report
flow that exposed false completion. The incomplete fixture is rejected without
Test, Feature, event, or outbox mutation. The complete fixture includes every
required passing report and reaches the explicit Feature completion review.
Independent human review approved the result before release preparation.

See `docs/dogfood/v0.4.3-cross-skill-integrity.md` for the commands, evidence,
limitations, and residual risks.

## Compatibility and boundaries

- DB schema remains **8**; there is no migration in v0.4.3.
- Runtime dependencies remain empty.
- Existing advisory projects remain compatible; new contract fields and
  approval provenance are additive where applicable.
- Report manifests declare the evidence universe. PCL does not prove that an
  external tool discovered every report it should have declared.
- Source references are audit claims, not identity authentication.
- No hosted backend, cloud sync, telemetry, paid service, automatic GitHub
  write, or external design-analysis implementation is added.
- Package, CLI, and MCP version metadata change from 0.4.2 to 0.4.3. The
  independently versioned Codex plugin manifest is unchanged.
- Task IDs shipped: 0150, 0151, 0152, 0153, and 0153b.
