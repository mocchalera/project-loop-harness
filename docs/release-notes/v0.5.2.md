# project-loop-harness v0.5.2

**Prepared:** 2026-07-18

**Status:** Published and publicly verified

**Scope:** reliability, target-bound operation, and evidence-backed harness
minimization

This release reduces setup and routing friction, makes proof health easier to
interpret, and packages the offline evaluation tools used to decide how much
harness structure to retain. It does not claim that external adoption has been
proven: the five-person first-use protocol and evaluator are ready, but no
participant outcomes have been collected.

## Highlights

### Safer adoption and installed-Skill maintenance

- Python and Node repositories receive config-ready, inspect-first
  initialization without executing project code or enabling unknown commands.
- `pcl init --repair-config` repairs only legacy empty command placeholders and
  preserves existing state and non-empty configuration.
- `pcl doctor` detects stale installed Project Loop Skills, while
  `pcl init --refresh-skill` previews and applies an explicit refresh with a
  SHA-256-addressed backup and audit event.
- The privacy-preserving five-person adoption protocol now has a deterministic,
  dependency-free offline evaluator and participant kit.

### Target-bound operation and clearer proof health

- `pcl next --target` and `pcl resume --target` use a shared Task/Goal resolver,
  so explicit current intent does not silently route to unrelated older work.
- Active blocking findings are separated from historical findings in status,
  audit, and rendered summaries.
- Copied Evidence audit findings classify operational impact instead of
  presenting every historical hash drift as a current blocker.
- Dashboard summary cards expose Goals, Tasks, active proof health, and
  historical findings without changing the underlying state model.

### Measured harness minimization

- A deterministic layered ablation evaluator compares direct execution,
  boundary-only guidance, state-only support, and the full harness across fixed
  baseline/treatment arms.
- The frozen experiment returned `modify`, not `proceed`; the proposed direct
  flow reduction was therefore not implemented. Boundaries, durable state,
  evidence, and completion gates remain in place while redundant prose is
  reduced only where runtime routing already covers it.
- CLI presentation and read-only guide/status/doctor/report handlers were split
  into smaller modules without changing their public behavior.

## Compatibility and boundaries

- DB schema remains **8**; no migration is added.
- Runtime dependencies remain empty and Python support remains 3.10-3.13.
- Core remains local-only and does not call an LLM, capture transcripts, upload
  telemetry, or add hosted/cloud state.
- Council remains opt-in and provider execution remains outside Core.
- Existing project configuration and installed Skills remain non-overwriting by
  default; repair and refresh require explicit flags.
